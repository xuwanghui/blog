鸿蒙开发

发表于 2025-03-10 更新于 2025-04-01 分类于 鸿蒙 阅读次数： Valine： 本文字数： 6.2k 阅读时长 ≈ 6 分钟

鸿蒙的诞生背景

为何华为要自研操作系统？

​ 在华为被美国商务部列入实体清单，面临谷歌GMS服务断供的困境下，华为急需一个能够替代安卓的操作系统，以确保其智能手机的海外市场不受影响。鸿蒙系统的诞生，正是在这样的背景下应运而生。它不仅是为了应对外部压力，更是华为长期以来坚持技术创新、追求自主可控战略的必然结果。那么，鸿蒙系统究竟是如何实现这一目标的？

​ 鸿蒙系统采用了分布式技术，能够实现跨设备的无缝协同，打破了传统操作系统局限于单一设备的界限。这意味着，用户可以在不同设备间自由切换，享受连续、一致的使用体验。这一特性不仅提升了用户体验，更为华为在物联网时代抢占先机提供了有力支撑。然而，面对已经根深蒂固的安卓和iOS系统，鸿蒙系统能否顺利打入市场，赢得用户的青睐？

鸿蒙的技术亮点

分布式架构与微内核

​ 提到鸿蒙系统，不得不提的就是其分布式架构和微内核设计。分布式架构使得鸿蒙系统能够轻松实现多设备间的互联互通，为用户带来前所未有的便捷体验。而微内核设计则大大提升了系统的安全性和稳定性，降低了被攻击的风险。

​ 具体来说，鸿蒙系统的微内核设计将系统服务与用户应用分离，使得系统更加轻量、高效。同时，微内核还具备自我修复能力，能够在遭受攻击时迅速恢复，确保系统的正常运行。这一设计无疑为鸿蒙系统赢得了不少技术爱好者的青睐。然而，技术上的优势能否转化为市场上的胜势，还需要看鸿蒙系统在实际应用中的表现。

鸿蒙的市场表现

从手机到物联网的全面布局

​ 自鸿蒙系统发布以来，华为已经将其应用于多款智能手机上，并取得了不错的市场反馈。然而，鸿蒙系统的野心显然不止于此。华为希望借助鸿蒙系统，在物联网领域实现全面布局，构建一个涵盖智能家居、智能穿戴、智能出行等多个场景的生态系统。

为了实现这一目标，华为正在积极与各大厂商合作，推动鸿蒙系统的普及和应用。同时，华为还在不断完善鸿蒙系统的生态体系，吸引更多开发者加入，共同打造丰富的应用场景。那么，鸿蒙系统能否在物联网领域大放异彩，成为国产操作系统的佼佼者？

鸿蒙的未来展望

挑战与机遇并存

尽管鸿蒙系统在技术上具备诸多优势，但在市场推广和生态建设方面仍面临不少挑战。一方面，安卓和iOS系统已经占据了智能手机市场的大部分份额，鸿蒙系统想要打破这一格局并不容易。另一方面，鸿蒙系统的生态建设还需要时间和资源的投入，短期内难以与成熟的安卓和iOS生态相媲美。

然而，挑战与机遇并存。随着物联网时代的到来，鸿蒙系统的分布式架构和微内核设计将为其带来前所未有的发展机遇。同时，华为在技术研发和市场推广方面的强大实力，也为鸿蒙系统的未来发展提供了有力保障。

那么，鸿蒙系统究竟能否成为国产操作系统的扛把子？这取决于华为能否在市场推广、生态建设等方面取得突破性进展，以及鸿蒙系统能否持续保持技术创新和用户体验优势。

HarmonyOs三大特性

硬件互助，资源共享

  • 硬件互助：鸿蒙系统可以将多个设备的硬件能力（如摄像头、麦克风、屏幕等）虚拟化为一个超级终端，用户可以根据需求调用不同设备的硬件能力。

  • 资源共享：鸿蒙系统支持多设备间的数据共享和任务协同，实现资源的无缝流转和高效利用。

硬件互助实现原理
  • 分布式软总线
    • 鸿蒙系统通过分布式软总线技术，将多个设备的硬件能力连接在一起。
    • 设备间通过低延迟、高带宽的通信协议进行交互。
  • 设备虚拟化
    • 将多个设备的硬件能力虚拟化为一个统一的资源池。
    • 用户无需关心具体设备，只需调用所需的能力。
资源共享实现原理
  • 分布式数据管理
    • 鸿蒙系统提供分布式数据库，支持多设备间的数据同步和共享。
    • 数据可以在设备间实时同步，确保一致性。
  • 分布式任务调度
    • 鸿蒙系统可以根据设备的能力和状态，智能分配任务。
    • 例如，手机可以将计算任务分发给平板或PC，提升效率。
应用场景
  • 多屏协同
    • 手机与平板、PC协同工作，共享屏幕和输入设备。
    • 例如，在平板上操作手机应用，或将手机内容投屏到电视。
  • 智能家居
    • 手机与智能家居设备（如智能音箱、摄像头）协同。
    • 例如，通过手机查看摄像头的实时画面，或通过音箱控制家电。
  • 健康管理
    • 手机与智能手表、体脂秤等设备协同。
    • 例如，健康数据在设备间同步，形成完整的健康报告。
一次开发，多端部署
什么是一次开发，多端部署？
  • 定义：开发者只需编写一次代码，即可将应用部署到多种设备上（如手机、平板、智能手表、智能家居设备等）。
  • 优势：
    • 降低开发成本，提高开发效率。
    • 统一用户体验，实现多设备协同。
实现原理
  • 分布式架构
    • 鸿蒙系统采用分布式架构，将不同设备的硬件能力和资源虚拟化为一个超级终端。
  • 统一开发框架
    • 提供统一的UI框架和API，支持多种设备的适配。
  • 自适应布局
    • 通过自适应布局和组件，确保应用在不同设备上都能良好显示。

统一OS，弹性部署
什么是统一OS， 弹性部署？
  • 统一OS：
    • 鸿蒙系统是一个统一的分布式操作系统，支持多种设备类型（如手机、平板、智能穿戴、智能家居等）。
    • 通过微内核架构和分布式技术，实现跨设备的无缝协同。
  • 弹性部署：
    • 开发者可以根据设备的硬件能力和使用场景，灵活地部署应用功能。
    • 应用可以按需加载模块，减少资源占用，提升运行效率。
统一OS的核心特性
  • 分布式架构
    • 通过分布式软总线技术，将多个设备的硬件能力虚拟化为一个超级终端。
    • 支持设备间的硬件互助和资源共享。
  • 微内核设计
    • 内核轻量化，安全性高，适合多种设备类型。
    • 支持动态加载和卸载功能模块。
  • 统一开发框架
    • 提供统一的API和UI框架，支持一次开发，多端部署。
弹性部署的实现方式
  • 模块化设计
    • 将应用功能拆分为多个独立模块，按需加载。
    • 示例：一个应用的核心功能可以在手机上运行，而扩展功能可以在平板上运行。
  • 动态加载
    • 根据设备的硬件能力和网络状态，动态加载或卸载功能模块。
    • 示例：在智能手表上只加载必要功能，减少资源占用。
  • 分布式数据管理
    • 通过分布式数据库，实现多设备间的数据同步和共享。
    • 示例：手机和平板可以实时同步应用数据。

弹性部署的智能家居控制应用

  • 场景描述
    • 开发一个智能家居控制应用，支持在手机、平板和智能音箱上运行。
    • 手机上提供完整功能，平板上提供部分功能，智能音箱上只提供语音控制功能。
  • 实现步骤
    1. 将应用功能拆分为多个模块（如控制模块、语音模块）。
    2. 根据设备类型动态加载模块。
    3. 使用分布式数据管理，实现多设备间的数据同步。

了解基本工程目录
工程级目录

  • AppScope中存放应用全局所需要的资源文件。
  • entry是应用的主模块，存放HarmonyOS应用的代码、资源等。
  • oh_modules是工程的依赖包，存放工程依赖的源文件。
  • build-profile.json5是工程级配置信息，包括签名、产品配置等。
  • hvigorfile.ts是工程级编译构建任务脚本，hvigor是基于任务管理机制实现的一款全新的自动化构建工具，主要提供任务注册编排，工程模型管理、配置管理等核心能力。
  • oh-package.json5是工程级依赖配置文件，用于记录引入包的配置信息。

在AppScope，其中有resources文件夹和配置文件app.json5，AppScope>resources>base中包含element和media两个文件夹

  • 其中element文件夹主要存放公共的字符串、布局文件等资源。
  • media存放全局公共的多媒体资源文件。
模块级目录

entry>src目录中主要包含总的main文件夹，单元测试目录ohosTest，以及模块级的配置文件。

  • main文件夹中，ets文件夹用于存放ets代码，resources文件存放模块内的多媒体及布局文件等，module.json5文件为模块的配置文件。
  • ohosTest是单元测试目录。
  • build-profile.json5是模块级配置信息，包括编译构建配置项。
  • hvigorfile.ts文件是模块级构建脚本。
  • oh-package.json5是模块级依赖配置信息文件。

进入src>main>ets目录中，其分为entryability、pages两个文件夹。

  • entryability存放ability文件，用于当前ability应用逻辑和生命周期管理。
  • pages存放UI界面相关代码文件，初始会生成一个Index页面。

resources目录下存放模块公共的多媒体、字符串及布局文件等资源，分别存放在element、media文件夹中。

app.json5

AppScope>app.json5是应用的全局的配置文件，用于存放应用公共的配置信息。

其中配置信息如下：

  • bundleName是包名。
  • vendor是应用程序供应商。
  • versionCode是用于区分应用版本。
  • versionName是版本号。
  • icon对应于应用的显示图标。
  • label是应用名。
module.json5

entry>src>main>module.json5是模块的配置文件，包含当前模块的配置信息。

其中module对应的是模块的配置信息，一个模块对应一个打包后的hap包，hap包全称是HarmonyOS Ability Package，其中包含了ability、第三方库、资源和配置文件。其具体属性及其描述可以参照下表1

表1 ：module.json5默认配置属性及描述

属性 描述
name 该标签标识当前module的名字，module打包成hap后，表示hap的名称，标签值采用字符串表示（最大长度31个字节），该名称在整个应用要唯一。
type 表示模块的类型，类型有三种，分别是entry、feature和har。
srcEntry 当前模块的入口文件路径。
description 当前模块的描述信息。
mainElement 该标签标识hap的入口ability名称或者extension名称。只有配置为mainElement的ability或者extension才允许在服务中心露出。
deviceTypes 该标签标识hap可以运行在哪类设备上，标签值采用字符串数组的表示。
deliveryWithInstall 标识当前Module是否在用户主动安装的时候安装，表示该Module对应的HAP是否跟随应用一起安装。- true：主动安装时安装。- false：主动安装时不安装。
installationFree 标识当前Module是否支持免安装特性。- true：表示支持免安装特性，且符合免安装约束。- false：表示不支持免安装特性。
pages 对应的是main_pages.json文件，用于配置ability中用到的page信息。
abilities 是一个数组，存放当前模块中所有的ability元能力的配置信息，其中可以有多个ability。

对于abilities中每一个ability的属性项，其描述信息如下表2

表2： abilities中对象的默认配置属性及描述

属性 描述
name 该标签标识当前ability的逻辑名，该名称在整个应用要唯一，标签值采用字符串表示（最大长度127个字节）。
srcEntry ability的入口代码路径。
description ability的描述信息。
icon ability的图标。该标签标识ability图标，标签值为资源文件的索引。该标签可缺省，缺省值为空。如果ability被配置为MainElement，该标签必须配置。
label ability的标签名。
startWindowIcon 启动页面的图标。
startWindowBackground 启动页面的背景色。
visible ability是否可以被其他应用程序调用，true表示可以被其它应用调用， false表示不可以被其它应用调用。
skills 标识能够接收的意图的action值的集合，取值通常为系统预定义的action值，也允许自定义。
entities 标识能够接收的Want的Action值的集合，取值通常为系统预定义的action值，也允许自定义。
actions 标识能够接收Want的Entity值的集合。
main_pages.json

src/main/resources/base/profile/main_pages.json文件保存的是页面page的路径配置信息，所有需要进行路由跳转的page页面都要在这里进行配置。
